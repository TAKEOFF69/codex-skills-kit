# Example: refactor (via codex-prompt)

## What this demonstrates

A cross-layer DB column rename – the case the plain Refactor scaffold is wrong for, because it assumes consumers are TypeScript imports and misses SQL `FROM` clauses, doc prose, and template interpolation. This example shows the **Cross-Layer Refactor Template**: a read-only **Phase 0 SCOPE AUDIT** that enumerates hits across all five layers (DB, app, docs, content, E2E baseline) via `rg` before any destructive step, dependency-leaves-first DB rewrites, and a byte-equal E2E re-check on an anchor entity. It also shows Directive mode (target structure is decided), a narrow risk gate, and the GO condition `Phase 1 scope ⊆ Phase 0 enumerated set`.

The task: rename the Postgres column `notes.is_pinned` to `notes.is_starred` across SQL, generated types, the API contract, and the export templates that interpolate the field name into user-facing output.

## The generated prompt

```markdown
# Codex Refactor: Rename notes.is_pinned → notes.is_starred (cross-layer)

Goal: Rename the `is_pinned` column on `notes` to `is_starred` everywhere it propagates – schema, migrations, generated types, API payloads, CLI export templates – with zero behavior change.

Success means:
  - `rg -n "is_pinned|isPinned|IsPinned" .` returns zero hits outside the deprecation shim noted below
  - The `notes` API returns `is_starred` with byte-identical values to the pre-refactor `is_pinned` on the anchor note
  - `acme export` output and the web client render the same star state as before, under the new field name
  - vitest + typecheck + build all green

Stop when: the rename lands across all five layers and the Phase 4 E2E re-check is byte-equal on two anchor notes, OR Phase 0 surfaces a hit outside the enumerated scope (then report and pause).

> Mode: Directive
> Reasoning: high
> Verification level: local tests + live DB check (local Postgres) + browser pass (web client)

## Context

`notes.is_pinned` is a boolean that has drifted in meaning – the product now calls this a "star," and the inconsistent name leaks into the API, the generated TS types, and the CLI export templates that interpolate the column name into a header. This is a pure rename (no semantic change). Because the identifier appears in SQL, docs, and a generated-content token, this is a cross-layer refactor, not a code-only one – the plain template's "consumers = TS imports" assumption would miss the SQL `FROM`, the migration history, and the export template.

Read `CONTRIBUTING.md` for project conventions (migration naming, type-gen command, test layout).

## Read these files FIRST

DB layer
1. `db/schema.sql` – canonical table definition
2. `db/migrations/` – existing migration history + naming pattern
3. `api/src/notes/queries.sql` – raw SQL the notes route runs

App layer
4. `api/src/notes/route.ts` – serializes the note payload
5. `api/src/db/types.gen.ts` – generated from schema (do NOT hand-edit)
6. `apps/web/src/lib/notes.ts` – client-side note model + the pin toggle

Content / template layer
7. `packages/cli/src/export.ts` – builds the export header from column names
8. `packages/cli/templates/note.md.hbs` – interpolates `{{is_pinned}}`

Test layer
9. `api/test/notes.test.ts` – payload-shape assertions
10. `packages/cli/test/export.test.ts` – export snapshot

## Assumptions

- The rename is semantic-neutral: every `true`/`false` value carries over unchanged. If Phase 0.E shows any value divergence, that is a data bug, not a rename target – stop and report.
- `api/src/db/types.gen.ts` is generated by `npm run gen:types`; it is regenerated, never hand-patched.
- One reversible deprecation shim is acceptable for one release: a generated SQL view `notes_legacy` exposing `is_pinned AS is_starred` for any external reader not in this repo. If you find no external reader in Phase 0, skip the shim and report that it was unnecessary.

## Invariant

Behavior unchanged – only the identifier changes. Star/pin truth values, API response shape (minus the field name), CLI export ordering, and the web toggle's effect all stay byte-for-byte equivalent. No new columns, no dropped rows, no copy changes beyond the field name itself.

## Risk Gate

Halt and report only for: (a) a Phase 0 hit outside the enumerated scope (the drift this template exists to catch), (b) a value divergence on the anchor note (would mean this is not a pure rename), (c) an external consumer of `is_pinned` that the deprecation shim cannot cover. Mechanical issues – a missed test fixture, a stale snapshot – are skip-and-log, fix-in-place, continue. Do NOT broaden the diff to "clean up" adjacent code.

## Repo mismatch stop condition

If any cited file/column/function is missing or differs (no `is_pinned` column, type-gen command renamed, no `templates/note.md.hbs`), stop and report the mismatch instead of inventing a target.

## Not in scope

- Renaming any other `is_*` boolean on `notes` (touch on report-only basis if you spot drift).
- The search index in `apps/web/src/lib/search.ts` – it indexes note bodies, not the pin flag; confirm in Phase 0 it has no hit, then leave it.
- Migrating historical export files already written to disk.

## Phase 0 – SCOPE AUDIT (read-only, gates all destructive phases)

Modify nothing. Produce one markdown report enumerating every hit from live state, not from this prompt's file list. The file list above is a starting point; `rg` is the source of truth.

### A. DB layer
- `rg -n "is_pinned" db/` across `schema.sql` + every migration
- Enumerate every view/table exposing the column. For each, capture its dependency graph (what reads it, what it reads) so DB renames run dependency-leaves-first.
- FLAG any hit not in the Read-these-files list – that is the drift this audit catches.

### B. App layer
- `rg -n "is_pinned|isPinned|IsPinned" api/ apps/` (snake + camel + pascal)
- Classify each hit: raw SQL, generated type, API serializer, React prop, test fixture.
- For any hit that surfaces to a user, note old vs new field name in the payload.

### C. Docs layer
- `rg -n "is_pinned" CONTRIBUTING.md README.md docs/`
- Note field-glossary or API-reference entries.

### D. Content / template layer
- `rg -n "is_pinned" packages/cli/`
- Confirm the Handlebars token `{{is_pinned}}` and any header label built from the column name.

### E. E2E baseline (anchor note)
Pick one anchor note id. Capture its pin value across the full chain pre-rename: `db` row → API JSON → `acme export` output line → web-client render. Record these byte-for-byte; Phase 4 re-asserts equality under the new name.

Report: tabulate A/B/C/D hits + the E baseline. **GO only if the Phase 1 rename scope is a subset of the Phase 0 enumerated set.** If Phase 0 found a hit the later phases don't cover, expand the plan or pause.

## Phase 1 – DB (destructive, dependency-leaves-first)
- New timestamped migration following the `db/migrations/` pattern: `ALTER TABLE notes RENAME COLUMN is_pinned TO is_starred;`
- If any view reads the column, recreate it in dependency order (leaves first) – Postgres rejects `ALTER` on a view's projected column; drop + recreate.
- Update `db/schema.sql` to match. Add the `notes_legacy` shim view only if Phase 0.A found an external reader.
- Snapshot `SELECT count(*) FILTER (WHERE is_starred) FROM notes;` and assert it equals the pre-rename pinned count.

## Phase 2 – App code
- Update `api/src/notes/queries.sql` and `route.ts` to the new column name.
- Run `npm run gen:types`; confirm `types.gen.ts` regenerated `is_starred` (do not hand-edit).
- Apply the rename in `apps/web/src/lib/notes.ts`. Typecheck + build green. Commit.

## Phase 3 – Docs + content
- Apply renames from Phase 0.C in `CONTRIBUTING.md` / docs.
- Update `packages/cli/src/export.ts` + `templates/note.md.hbs` (`{{is_pinned}}` → `{{is_starred}}`).
- Regenerate the export snapshot fixture; confirm the diff is field-name-only, not value-only.

## Phase 4 – E2E verification
- Re-run Phase 0.E on the anchor note + one second note. Assert byte-equal pin/star values under the new field name across `db → API → export → web render`.
- Browser spot-check: the star toggle in the web client still flips and persists.

## Constraints
- Behavior stays identical – this is structure/identifier only. No semantic change rides along.
- Stay within the enumerated scope. A correctness need for an out-of-scope file → report and pause, do not silently expand.
- Regenerate `types.gen.ts` via `npm run gen:types`; never hand-edit generated output.
- Rename a column via a new migration, never by editing a committed migration.
- Use en dashes (–) in any prose you write; never em dashes or double hyphens.
- DO NOT deploy. Commit + push only.

## Done when
- All five Phase 0 layers enumerated and reported; Phase 1 scope confirmed ⊆ Phase 0 set
- `rg -n "is_pinned|isPinned|IsPinned" .` returns zero (or only the documented `notes_legacy` shim)
- `npm run gen:types` produces `is_starred`; no hand-edits in `types.gen.ts`
- `npm test` (vitest) passes – including the regenerated export snapshot
- `npm run typecheck` + `npm run build` green
- Phase 4 byte-equal on two anchor notes; web star toggle verified in browser

### Manual verification

rg -n "is_pinned|isPinned" api/ apps/ packages/ db/

```

## Why it's shaped this way

- **It uses the Cross-Layer template, not plain Refactor, because the identifier crosses layers.** The skill's heuristic is explicit: an identifier that appears in SQL, a doc, or a generated-content token is cross-layer. Plain Refactor enumerates consumers as TS imports and would silently miss `queries.sql`'s `FROM`, the migration history, and the `{{is_pinned}}` Handlebars token in the export template.
- **Phase 0 is read-only and gates every destructive phase, enumerating from `rg` rather than the prompt's own file list.** A curated file list drafted from a pre-audit reliably misses adjacent writers; the SCOPE AUDIT exists precisely to surface hits the prompt-writer didn't know about, which is why its GO condition is `Phase 1 scope ⊆ Phase 0 set`.
- **The E2E baseline (Phase 0.E) plus the byte-equal Phase 4 re-check are the proof the rename was semantic-neutral.** Capturing the anchor note's value across `db → API → export → render` before the change turns "behavior unchanged" from a claim into a checkable assertion – a value divergence would prove this was never a pure rename.
- **Mode is `[LOCK]` Directive: the target name and the layer order are decided, so the prompt ships exact steps.** The investigative room is deliberately fenced into Phase 0's enumeration step; the rename itself has one right answer and no benefit from letting the agent explore.
- **The risk gate is narrow on purpose – three real halt conditions, skip-and-log for everything else.** It halts only for out-of-scope drift, value divergence, or an uncoverable external consumer; a missed snapshot is fixed in place and the work continues, so a mechanical hiccup never strands a correct multi-layer rename mid-flight.
