# Example: audit (via codex-prompt)

## What this demonstrates

A read-only review of the Acme Notes API auth layer against the eight common access-control bugs, generated by the `codex-prompt` skill's Audit scaffold. It shows the load-bearing Audit-category features: **Investigative mode** (PASS/FAIL gates frame the destination, methodology stays open), a **prescribed output format** (fixed PASS/FAIL table columns + per-finding rubric), the **internal-sources-first** ordering, the **read-only / DO NOT edit** constraint, **HEAD re-anchoring** at start and end, and **streaming the report to disk** so a long pass survives the context ceiling.

## The generated prompt

```markdown
# Codex Audit: API auth layer – access-control bug sweep

Goal: produce a PASS/FAIL verdict on the Acme Notes API auth layer for the eight common access-control bug classes, with a per-finding rubric and reproduction note, without changing any code.

Success means:
  - Every one of the 8 bug classes has a PASS or FAIL row with cited file:line evidence.
  - Each FAIL names the vulnerable path, the missing/weak check, and a concrete request that exploits it.
  - Internal sources (route handlers, middleware, the authz helper, the session store) are read before any dependency or upstream-library behavior is assumed.

Stop when: the verdict table is written to disk for all 8 classes and the HEAD re-anchor check has run. No code is edited.

> Mode: Investigative
> Reasoning: high
> Verification level: code read

## Context
Acme Notes is a self-hostable markdown notes app: a Next.js (App Router) web client, a Node/TypeScript API, and Postgres. The API gates note read/write/share/export on a session cookie resolved to a `userId`, then checks ownership/collaboration per note. We want an honest read-only audit of that gate before the next release – no fixes in this pass, findings only. Treat every authorization decision as guilty until the code proves it innocent.

Read `CONTRIBUTING.md` for repo conventions and the test-run command.

## Read these files FIRST (internal sources before anything external)
1. `api/src/auth/session.ts` – cookie → userId resolution
2. `api/src/auth/authorize.ts` – per-resource authz helper
3. `api/src/middleware.ts` – global request guard / route matcher
4. `api/src/notes/route.ts` – note read/write/delete handlers
5. `api/src/notes/[id]/share/route.ts` – collaborator + share-link handlers
6. `api/src/notes/[id]/export/route.ts` – export handler
7. `api/src/db/queries.ts` – note/owner/collaborator SQL (check the WHERE clauses)
8. `api/test/auth.test.ts` – existing auth coverage (find the gaps it does NOT assert)

## Anchor HEAD
Run `git rev-parse HEAD` and record it as `HEAD_START` in the report header. Run it again at the end as `HEAD_END`; if they differ, re-verify every file:line finding against the new HEAD before issuing the verdict.

## The 8 access-control bug classes to check
1. **Missing authn** – an endpoint that resolves no session yet returns note data.
2. **Missing authz (IDOR)** – an authenticated user reaching another user's note by id with no ownership/collaborator check.
3. **Broken object-ownership scoping** – a query that filters by `id` but not by `userId`/collaborator, so the WHERE clause leaks rows.
4. **Verb/method confusion** – read allowed but write/delete/share reusing the same (read-only) check, or a check applied to GET but skipped on mutation verbs.
5. **Trusting client-supplied identity** – `userId`/`ownerId`/role read from the request body, query string, or a header instead of the resolved session.
6. **Share/collaborator escalation** – a collaborator (or share-link holder) able to delete the note, change owner, or add further collaborators beyond their grant.
7. **Export bypass** – the export path reaching note content through a code path that skips the authz helper the read path uses.
8. **Middleware gaps** – routes not matched by `middleware.ts` (matcher regex, route-group, or a handler mounted outside the guarded tree) that self-authorize inconsistently.

## Method (Investigative)
For each class: locate where the decision is (or should be) made, read the actual check, then try to construct a request that defeats it from the code as written. Prefer evidence from the route handler + the SQL it runs over assumptions about what a framework "probably" does. When a check lives in a shared helper, confirm every in-scope handler actually calls it – a helper that exists is not a helper that runs.

## Output format (write exactly this, stream to disk as you go)
Write to `audit-reports/api-auth-access-control.md`. Append each row as you finish it; do NOT buffer the whole report in memory and write once at the end.

Header block:
```
HEAD_START: <sha>
HEAD_END: <sha>   (filled at completion)
Verification level: code read
Scope: api/src/auth, api/src/notes, api/src/middleware.ts, api/src/db/queries.ts
```

Verdict table (one row per class):

| # | Bug class | Verdict | Evidence (file:line) | Exploit / why-safe (one line) |
|---|-----------|---------|----------------------|-------------------------------|

Then a `## Findings` section, one block per FAIL only, each scored on this rubric:
- **Severity:** CRITICAL (cross-tenant data read/write with no auth) / HIGH (cross-tenant via authenticated user) / MEDIUM (escalation within a granted relationship) / LOW (defense-in-depth gap, not directly exploitable)
- **Path:** the exact request (method + route + minimal body/params) that exploits it
- **Root:** the missing or weak check, quoted from the file
- **Confidence:** confirmed-by-reading-code / suspected-needs-runtime-check

Verdict rule: a class is **PASS** only when a correct check is present AND reached on every in-scope path. "Probably fine" is a FAIL with Confidence = suspected.

## Assumptions
- The session cookie → `userId` resolution in `api/src/auth/session.ts` is the single source of caller identity; any handler deriving identity elsewhere is a class-5 finding, not an alternative design.
- Collaborator grants are read-or-comment unless the schema in `queries.ts` proves a write/admin grant exists; assume least privilege when ambiguous and flag the ambiguity rather than guessing.

## Invariant
Read-only audit: no source file, migration, test, or config is modified. The working tree is identical at `HEAD_END` to `HEAD_START`.

## Risk Gate
This is a non-destructive read pass; there is no destructive halt. If the auth layer's structure differs materially from this prompt (e.g. authz is enforced in the DB via row-level security rather than in `authorize.ts`), record that as a finding, adjust the method to audit the real enforcement point, and continue – do not stop.

## Repo mismatch stop condition
If a listed file/route/export is absent (e.g. no `api/src/auth/authorize.ts`, or notes are not under `api/src/notes/`), report the mismatch with what you found instead, then audit the real structure rather than aborting.

## Not in scope (report-only)
- The Next.js web client (`apps/web/**`) – note any auth check duplicated client-side, but the gate under test is the API.
- The `acme` CLI (`packages/cli/**`) – flag only if it bypasses the API and talks to Postgres directly.
- Rate limiting, CSRF, and input validation – out of this pass unless they are the only thing standing between a caller and another user's note.

## Constraints
- DO NOT edit files. No commits, no migrations, no test edits. This pass produces one report and nothing else.
- Internal sources first: read the handlers, middleware, authz helper, and SQL before reasoning about framework or library defaults.
- Cite only file:line you have actually read; an uncited claim is a suspected finding, not a confirmed one.
- Verify via: reading the code path end to end (request → middleware → handler → authz helper → SQL). Reproduce exploit requests on paper from the code; do not run them against a live instance.
- Stream the report to `audit-reports/api-auth-access-control.md` incrementally so a long pass survives the context ceiling.

## Done when
- `audit-reports/api-auth-access-control.md` exists with HEAD_START + HEAD_END, the 8-row verdict table, and a Findings block for every FAIL.
- `git status --porcelain` shows only the new report file – no source changes.
- `HEAD_END` recorded; if it differs from `HEAD_START`, every finding re-verified against the new HEAD before the verdict stands.
```

## Why it's shaped this way

- **Investigative mode, not Directive** – the Audit default. We know the destination (a PASS/FAIL verdict on 8 known classes) but not where the bugs live, so the prompt ships gates and method and leaves Codex room to trace each path. Prescribing exact steps here would foreclose the unexpected route handler that self-authorizes.
- **The output format is [LOCK], the methodology is [FORK].** Fixed table columns, a fixed severity rubric, and a hard PASS-means rule make the result comparable and reviewable; how Codex hunts each bug is deliberately left open. Locking the wrong half (steps) and forking the right half (format) is the classic audit failure.
- **Internal sources first** is ordered explicitly because the cheapest false-negative in an auth audit is assuming a framework or library enforces something the handler actually skips. The own-code paths get read before any upstream behavior is trusted.
- **Read-only is stated three ways** – `DO NOT edit files`, the Invariant, and a `git status --porcelain` check in Done-when – because an audit that quietly "fixes while it's in there" stops being an audit and loses the clean before/after the reviewer needs.
- **HEAD re-anchor + streaming writes** are incident-derived guards: a long pass can outrun the commit it started on (stale findings) or die before a single buffered Write (lost report). Anchoring both ends and appending each row as it lands makes the pass restart-safe and its findings trustworthy.
```
